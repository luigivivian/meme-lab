## Summary

This phase adds burned-in text overlays (meme phrase + watermark) to Sora 2 generated videos using FFmpeg's `drawtext` filter. The core challenge is faithfully porting the Pillow-based text rendering from `image_maker.py` (white text, black stroke, shadow, word-wrap, watermark) into FFmpeg filter chain parameters. FFmpeg's `drawtext` supports all the needed styling (`borderw`, `bordercolor`, `shadowcolor`, `shadowx/shadowy`, `alpha`) and accepts the same `.ttf` font files the project already uses in `assets/fonts/`.

The primary technical risk is word-wrapping: FFmpeg's `drawtext` has no built-in max-width word-wrap -- it renders text as-is with newline support. The solution is to **pre-wrap text in Python using Pillow's font metrics** (reusing `_wrap_text()` from `image_maker.py`) and pass the pre-wrapped text with embedded `\n` characters to FFmpeg. This guarantees pixel-identical line breaks between image and video text. Three animation modes (static, fade-in, typewriter) are achievable purely through `drawtext`'s expression language using the `alpha` and `enable` parameters with time-based conditionals.

**Primary recommendation:** Use `subprocess` directly (not `ffmpeg-python` wrapper) for FFmpeg invocation. The `ffmpeg-python` library (v0.2.0, last release 2019) is effectively unmaintained with 47 open PRs and no new releases in 7 years. Direct subprocess gives full control over the `drawtext` filter chain, avoids Windows path escaping issues in the wrapper, and aligns with the project's existing `asyncio.to_thread()` pattern for CPU-bound operations. FFmpeg must be installed on the system (not currently present -- `winget install ffmpeg` or `choco install ffmpeg`).

### Pattern 1: LegendRenderer (FFmpeg Filter Chain Builder)
**What:** A stateless class that constructs FFmpeg `drawtext` filter strings and executes them via subprocess. Handles text wrapping, font resolution, mode-specific alpha expressions, and Windows path escaping.
**When to use:** Any time text needs to be burned into a video file.
**Example:**
```python
# Source: Synthesized from FFmpeg drawtext documentation + project patterns
import subprocess
import tempfile
from pathlib import Path
from PIL import ImageFont

class LegendRenderer:
    """Renders text overlays on video using FFmpeg drawtext filter."""

    def __init__(self, font_path: str, font_size: int = 48):
        self._font_path = font_path
        self._font_size = font_size
        self._pil_font = ImageFont.truetype(font_path, font_size)

    def render(
        self,
        video_path: str,
        output_path: str,
        phrase: str,
        watermark: str = "",
        mode: str = "static",
        video_width: int = 1080,
        video_height: int = 1350,
    ) -> str:
        """Burn text overlay into video. Returns output_path on success."""
        # Pre-wrap using Pillow font metrics (identical to image_maker.py)
        margin = 80
        max_width = video_width - (margin * 2)
        lines = self._wrap_text(phrase.upper(), max_width)
        wrapped_text = "\n".join(lines)

        # Build filter chain
        filters = self._build_phrase_filter(
            wrapped_text, mode, video_width, video_height,
        )
        if watermark:
            filters += "," + self._build_watermark_filter(
                watermark, video_width, video_height,
            )

        cmd = [
            "ffmpeg", "-y",
            "-i", video_path,
            "-vf", filters,
            "-c:v", "libx264", "-crf", "18",
            "-c:a", "copy",
            "-movflags", "+faststart",
            output_path,
        ]
        subprocess.run(cmd, check=True, capture_output=True)
        return output_path
```

**What:** Write the phrase to a temporary file and use `textfile=` instead of `text=` to avoid shell/FFmpeg double-escaping nightmares on Windows.
**When to use:** Always -- especially for user-generated text with special characters.

### Pattern 3: Mode-Specific Alpha Expressions
**What:** Each animation mode uses different `alpha` and/or `enable` expressions in drawtext.
**When to use:** Controlled by `VIDEO_LEGEND_MODE` config or per-request API param.
**Example:**
```python
# STATIC: full opacity, always visible
alpha_expr = "1"

# FADE-IN: 0.5s fade-in, then stays visible (per D-06)
alpha_expr = "if(lt(t,0.5),t/0.5,1)"

# TYPEWRITER: reveal characters progressively at ~30 chars/sec (per D-07)
# Use multiple drawtext filters, one per line, with enable timing
# Line 1 appears at t=0, Line 2 after Line 1 completes, etc.
chars_per_sec = 30
line_start = 0.0
for i, line in enumerate(lines):
    line_duration = len(line) / chars_per_sec
    # Each line fades in over its duration
    enable_expr = f"gte(t,{line_start})"
    # Reveal chars: use text[0:min(len, floor((t-start)*cps))]
    # ... via textfile + reload approach or chained drawtext per char group
    line_start += line_duration + 0.1  # small gap between lines
```

    """Burns text legend onto generated videos."""

    def __init__(self, renderer: LegendRenderer | None = None):
        self._renderer = renderer or LegendRenderer(font_path=_resolve_font())

### Anti-Patterns to Avoid
- **Building FFmpeg commands via string concatenation:** Use lists for `subprocess.run()` args to avoid injection and escaping issues. Never use `shell=True`.
- **Using `ffmpeg-python` for drawtext on Windows:** The wrapper has known issues with path escaping (colon in `C:\`) and special characters in text. Direct subprocess is safer.
- **Rendering text directly in FFmpeg without pre-wrapping:** FFmpeg's drawtext has no word-wrap. Text will overflow horizontally without Python-side pre-wrapping.
- **Hardcoding font path:** Always resolve via `_load_font()` pattern from `image_maker.py` -- check `assets/fonts/*.ttf` first, fall back to system fonts.
- **Blocking the event loop with FFmpeg:** FFmpeg is CPU-bound and can take seconds. Always use `asyncio.to_thread()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Video text rendering | Custom frame-by-frame Python overlay (OpenCV/Pillow per frame) | FFmpeg drawtext filter | FFmpeg is hardware-accelerated, single-pass, handles encoding/muxing. Frame-by-frame in Python would be 100x slower. |
| Word-wrap for video | New word-wrap algorithm based on character counts | Pillow `ImageFont.getbbox()` with existing `_wrap_text()` from image_maker.py | Font metrics are not monospace -- must use actual glyph widths. Pillow has the exact same font loaded. |
| FFmpeg path escaping on Windows | Manual string replace for colons/backslashes | `textfile` param (write text to temp file) + forward-slash paths | FFmpeg on Windows accepts forward slashes. Textfile avoids text escaping entirely. |
| Video metadata extraction | Parsing ffprobe output manually | `ffprobe -v quiet -print_format json -show_streams` | Standard approach, JSON output is trivially parseable |
| H.264 encoding settings | Custom bitrate/quality calculations | `-crf 18 -preset medium -movflags +faststart` | CRF 18 is visually lossless, faststart enables progressive download/streaming |

**Key insight:** FFmpeg's drawtext filter handles ALL the text rendering (font rasterization, stroke, shadow, alpha blending) in a single video pass. The Python code only needs to: (1) pre-wrap text using Pillow metrics, (2) construct the filter string, (3) invoke FFmpeg subprocess. No video frame manipulation in Python.

## Common Pitfalls

### Pitfall 1: Windows Font Path Escaping in drawtext
**What goes wrong:** FFmpeg's drawtext `fontfile` parameter chokes on Windows paths containing colons and backslashes (`C:\Users\...\assets\fonts\Cinzel.ttf`).
**Why it happens:** FFmpeg uses `:` as parameter separator in filter strings and `\` as escape character.
**How to avoid:** Use forward slashes in all paths passed to FFmpeg (`C:/Users/.../assets/fonts/Cinzel.ttf`). For the `fontfile` parameter, escape the colon in the drive letter: `C\:/Users/...` or better yet, use a relative path from the working directory if possible.
**Warning signs:** FFmpeg errors like "Option fontfile not found" or "Invalid option" or truncated paths.

### Pitfall 2: Text Escaping in drawtext text Parameter
**What goes wrong:** Special characters in meme phrases (quotes, colons, backslashes, percent signs) break the FFmpeg filter string parser.
**Why it happens:** FFmpeg has multiple levels of escaping: filter option level, filtergraph level, and shell level.
**How to avoid:** Use `textfile=` parameter instead of `text=`. Write the phrase to a temporary UTF-8 text file and reference it. This completely bypasses text escaping.
**Warning signs:** Garbled text output, FFmpeg parse errors, missing characters.

### Pitfall 3: No Word-Wrap in drawtext
**What goes wrong:** Long phrases extend beyond the video frame horizontally, with no visible text or text clipped at edges.
**Why it happens:** FFmpeg's drawtext does not have a max-width or word-wrap parameter.
**How to avoid:** Pre-wrap text in Python using Pillow's `ImageFont.getbbox()` to measure actual glyph widths, then insert `\n` at wrap points before passing to FFmpeg. Port the exact `_wrap_text()` function from `image_maker.py`.
**Warning signs:** Single-line text extending off-screen, different wrap points than image text.

### Pitfall 4: FFmpeg Binary Not Found
**What goes wrong:** `subprocess.run(["ffmpeg", ...])` raises `FileNotFoundError` because FFmpeg is not installed or not on PATH.
**Why it happens:** FFmpeg is a system dependency, not a pip package. Unlike Python libraries, it must be installed separately.
**How to avoid:** Check FFmpeg availability at startup/first use with `shutil.which("ffmpeg")`. Provide clear error message: "FFmpeg not found. Install via: winget install Gyan.FFmpeg". Make `VIDEO_LEGEND_ENABLED` default to `false` so missing FFmpeg doesn't break the pipeline.
**Warning signs:** Import-time errors, pipeline crashes on first video with legends.

### Pitfall 5: Typewriter Mode Complexity
**What goes wrong:** Implementing true character-by-character reveal requires either (a) chaining hundreds of drawtext filters (one per character) or (b) complex expression-based text slicing which FFmpeg does not natively support.
**Why it happens:** FFmpeg's drawtext cannot dynamically substring the text parameter based on time.
**How to avoid:** Implement typewriter as **word-by-word** reveal (not char-by-char) using chained drawtext filters -- one per word or line. Each filter has an `enable='gte(t,X)'` parameter. This gives a natural-looking typewriter effect with manageable filter complexity (5-10 filters instead of 60+).
**Warning signs:** Extremely long filter chains, FFmpeg memory issues, slow rendering.

### Pitfall 6: Instagram Safe Zone Miscalculation
**What goes wrong:** Text is positioned at 80% vertical but Instagram Reels UI covers the bottom ~25% of the frame.
**Why it happens:** Videos generated at 4:5 (1080x1350) are displayed differently in Reels feed vs. feed post. In Reels mode, UI overlays cover bottom area.
**How to avoid:** Per D-12, apply 10% vertical safe margin. For 1080x1350 video, this means text should not extend below `y = 1350 * 0.90 = 1215px`. Calculate: `text_y = h * 0.80 - text_h/2` but clamp so `text_y + text_h < h * 0.90`.
**Warning signs:** Text hidden behind Instagram like/comment/share buttons or username overlay.

# Source: Project image_maker.py watermark pattern + FFmpeg drawtext docs
# Matches image_maker.py: 22pt font, bottom-right corner, semi-transparent gold

def _build_watermark_filter(
    watermark_text: str,
    font_path: str,
    wm_font_size: int = 22,

    """Build drawtext filter for watermark."""

### Windows Path Escaping Helper
```python
# Source: FFmpeg drawtext docs on Windows path handling
import platform

def _escape_ffmpeg_path(path: str) -> str:
    """Escape a file path for use in FFmpeg filter strings.

    On Windows, converts backslashes to forward slashes and
    escapes the colon in drive letters (C: -> C\\:).
    """
    p = str(path).replace("\\", "/")
    if platform.system() == "Windows" and len(p) >= 2 and p[1] == ":":
        p = p[0] + "\\:" + p[2:]
    return p
```

### Word-Wrap Port from image_maker.py
```python
# Source: src/image_maker.py:_wrap_text() -- ported for video use
from PIL import ImageFont

def wrap_text_for_video(
    text: str,
    font: ImageFont.FreeTypeFont,
    max_width: int,
) -> list[str]:
    """Wrap text using Pillow font metrics (identical to image_maker.py).

    Uses actual glyph widths, not character counts, for accurate wrapping.
    """
    words = text.split()
    lines = []
    current_line = ""

    for word in words:
        test_line = f"{current_line} {word}".strip()
        bbox = font.getbbox(test_line)
        text_width = bbox[2] - bbox[0]

        if text_width <= max_width:
            current_line = test_line
        else:
            if current_line:
                lines.append(current_line)
            current_line = word

    if current_line:
        lines.append(current_line)

    return lines if lines else [text]
```

### Subprocess Invocation with asyncio.to_thread()
```python
# Source: Project convention from kie_client.py, caption_worker.py
import asyncio
import subprocess

async def render_legend_async(
    video_path: str,
    output_path: str,
    filter_chain: str,
) -> str:
    """Run FFmpeg legend render in thread pool (non-blocking)."""
    def _run():
        cmd = [
            "ffmpeg", "-y",
            "-i", video_path,
            "-vf", filter_chain,
            "-c:v", "libx264",
            "-crf", "18",
            "-preset", "medium",
            "-c:a", "copy",
            "-movflags", "+faststart",
            output_path,
        ]
        result = subprocess.run(
            cmd,
            capture_output=True,
            text=True,
            timeout=120,  # 2 min timeout for 10-15s video
        )
        if result.returncode != 0:
            raise RuntimeError(f"FFmpeg failed: {result.stderr[:500]}")
        return output_path

    return await asyncio.to_thread(_run)
```

## Open Questions

1. **Video resolution from Sora 2**
   - What we know: Kie.ai generates "portrait" (4:5) aspect ratio videos. Image dimensions are 1080x1350.
   - What's unclear: Exact pixel resolution of Sora 2 output (could be 720x900, 1080x1350, or higher). Text positioning depends on actual resolution.
   - Recommendation: Probe video with `ffprobe` at render time to get actual width/height. Use proportional positioning (`h*0.80`) not absolute pixels.

2. **Typewriter granularity: char-by-char vs word-by-word**
   - What we know: D-07 says "~30 chars/sec" and "char-by-char" in phase description.
   - What's unclear: True char-by-char requires complex FFmpeg expressions or many chained filters. Word-by-word is simpler and looks natural.
   - Recommendation: Implement as **line-by-line reveal with per-line fade-in** (each line appears 0.3s after previous). This looks like typewriter, is simple to implement (one drawtext per line with `enable` timing), and avoids char-level complexity. If user requests true char-by-char later, it can be added as a separate mode.

3. **Audio stream handling**
   - What we know: Sora 2 videos may or may not have audio tracks.
   - What's unclear: Whether `-c:a copy` fails if no audio stream exists.
   - Recommendation: Use `-c:a copy` with error handling, or probe for audio first. Add `-an` flag if no audio detected.
