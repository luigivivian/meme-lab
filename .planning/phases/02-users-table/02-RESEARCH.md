**Researched:** 2026-03-23

## Summary

Phase 2 creates a `users` table in MySQL via Alembic migration, adds a nullable `user_id` FK to the existing `characters` table, and extends the seed script to create an admin user. The codebase already has 11 ORM models, 6 migrations (including one orphaned autogenerated migration), and a well-established pattern for column definitions, indexes, and relationships.

The technical scope is narrow and well-understood: one new model class, one new migration file, one seed function addition, and one FK column added to an existing table. The only external dependency to install is `bcrypt` for password hashing in the seed script (needed now, not Phase 3, because the admin user password must be hashed at seed time).

**Primary recommendation:** Follow the exact column definition patterns from existing models (Mapped/mapped_column, server_default for simple types, ORM default for JSON/Text), use bcrypt 5.0.0 directly (no passlib wrapper needed), and number the migration `006` chaining from `005`.

|------------|-----------|----------|
| bcrypt (direct) | passlib[bcrypt] | passlib adds unnecessary abstraction; bcrypt 5.0 API is simple enough: `bcrypt.hashpw()` / `bcrypt.checkpw()` |

**Discretion decision: Use bcrypt 5.0.0 directly.** It is the simplest option, matches AUTH-01 requirement text, and avoids adding passlib as an unnecessary dependency layer. The API is two functions: `bcrypt.hashpw(password, bcrypt.gensalt())` and `bcrypt.checkpw(password, hashed)`.

### Pattern 2: Alembic Migration (from existing codebase)
**What:** Manual migration files with sequential numbering (001-005), pure DDL in upgrade/downgrade.
**When to use:** The users table migration.
**Example:**
```python
# Source: src/database/migrations/versions/004_add_scheduled_posts.py
revision: str = '006'
down_revision: Union[str, None] = '005'

def upgrade() -> None:
    op.create_table("users", ...)
    op.add_column("characters", sa.Column("user_id", sa.Integer(), sa.ForeignKey("users.id"), nullable=True))
    op.create_index("idx_characters_user_id", "characters", ["user_id"])

def downgrade() -> None:
    op.drop_index("idx_characters_user_id", table_name="characters")
    op.drop_column("characters", "user_id")
    # drop indexes first, then table
    op.drop_table("users")
```

### Pattern 3: Seed Script Extension (from existing codebase)
**What:** Async seed functions in seed.py, idempotent (check-before-insert).
**When to use:** Admin user creation.
**Example:**
```python
# Source: src/database/seed.py (follows seed_characters pattern)
async def seed_admin_user(session):
    """Cria admin user se ADMIN_EMAIL e ADMIN_PASSWORD estiverem no env."""
    from sqlalchemy import select
    import bcrypt

    admin_email = os.environ.get("ADMIN_EMAIL", "").strip().lower()
    admin_password = os.environ.get("ADMIN_PASSWORD", "").strip()

    if not admin_email or not admin_password:
        print("  [SKIP] ADMIN_EMAIL ou ADMIN_PASSWORD nao definidos no env.")
        return

    existing = await session.execute(
        select(User).where(User.email == admin_email)
    )
    if existing.scalar_one_or_none():
        print(f"  [SKIP] Admin '{admin_email}' ja existe no banco.")
        return

    hashed = bcrypt.hashpw(admin_password.encode("utf-8"), bcrypt.gensalt())
    user = User(
        email=admin_email,
        hashed_password=hashed.decode("utf-8"),
        role="admin",
        is_active=True,
    )
    session.add(user)
    await session.flush()
    print(f"  [OK] Admin '{admin_email}' criado (id={user.id}).")
```

### Pattern 4: Character FK Addition
**What:** Add nullable `user_id` FK to characters table + bidirectional relationship.
**Discretion decision: Add back-populates.** The Character model already uses `back_populates` for all its relationships. Adding `owner: Mapped[Optional["User"]] = relationship(back_populates="characters")` follows the established pattern and enables queries like `character.owner` in Phase 4+.

### Anti-Patterns to Avoid
- **server_default on Text/JSON columns:** MySQL does not support default values on TEXT/JSON columns. Use ORM `default=` only. The codebase has this pattern documented in comments.
- **Data operations in migration:** D-04 explicitly says seed admin goes in seed.py, not in the migration. Keep migration as pure DDL.
- **Forgetting email normalization:** D-08 requires lowercase normalization before INSERT. The seed script must `.lower()` the email. Model-level enforcement can use a `@validates` decorator or simply rely on app-level normalization (Phase 3 Pydantic).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Password hashing | Custom hash function | bcrypt.hashpw() / bcrypt.checkpw() | Timing-safe comparison, configurable work factor, industry standard |
| Migration management | Raw ALTER TABLE SQL | Alembic op.create_table / op.add_column | Reversible, version-tracked, integrates with ORM metadata |
| Unique email constraint | Application-level check | DB unique index + IntegrityError handling | Race condition free, enforced at DB level |

## Common Pitfalls

### Pitfall 1: Orphaned Migration Chain
**What goes wrong:** There is already an orphaned migration `ee583b64523f_add_rendering_column_to_characters.py` that revises `001` (competing with `002`). New migration `006` must chain from `005`, not from this orphan.
**Why it happens:** Alembic autogenerate created a migration that was not integrated into the sequential chain.
**How to avoid:** Explicitly set `down_revision = '005'` in the new migration. Do NOT use `alembic revision --autogenerate` -- write manually like all other migrations in this project.
**Warning signs:** `alembic heads` showing multiple heads.

### Pitfall 2: bcrypt bytes vs string
**What goes wrong:** bcrypt.hashpw() returns `bytes`, but SQLAlchemy String/Text column expects `str`.
**Why it happens:** bcrypt operates on bytes internally.
**How to avoid:** Always `.decode("utf-8")` the hash before storing, and `.encode("utf-8")` from DB before checking.
**Warning signs:** TypeError on insert or comparison.

### Pitfall 3: Missing User import in seed.py
**What goes wrong:** seed.py currently imports Character, CharacterRef, Theme. After adding User model, must also import User.
**Why it happens:** Easy to forget when extending existing code.
**How to avoid:** Add User to the import statement at top of seed.py.

### Pitfall 4: Foreign Key ordering in migration downgrade
**What goes wrong:** Dropping the `users` table before removing the FK from `characters` causes a constraint error.
**Why it happens:** MySQL enforces FK constraints during DDL.
**How to avoid:** In `downgrade()`, first drop the FK index and column from `characters`, then drop the `users` table.

### Pitfall 5: hashed_password column too short
**What goes wrong:** bcrypt hashes are 60 characters. Using String(60) works but leaves no room for future algorithms (argon2 hashes are ~97+ chars).
**Why it happens:** Exact-fit column sizing.
**How to avoid:** Use `String(200)` for hashed_password, matching the generous sizing pattern in this codebase.

## Code Examples

### User Model Complete Definition
```python
# Follows Character model pattern from src/database/models.py
from typing import Optional
from sqlalchemy import Boolean, ForeignKey, Integer, String, Text
from sqlalchemy.orm import Mapped, mapped_column, relationship
from src.database.base import Base, TimestampMixin

class User(TimestampMixin, Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    email: Mapped[str] = mapped_column(String(255), unique=True, nullable=False)
    hashed_password: Mapped[str] = mapped_column(String(200), nullable=False)
    role: Mapped[str] = mapped_column(String(20), default="user", server_default="user")
    is_active: Mapped[bool] = mapped_column(Boolean, default=True, server_default="1")
    display_name: Mapped[Optional[str]] = mapped_column(String(200), nullable=True)

    # API keys — plaintext, nullable (D-01, D-02)
    gemini_free_key: Mapped[Optional[str]] = mapped_column(Text, nullable=True)
    gemini_paid_key: Mapped[Optional[str]] = mapped_column(Text, nullable=True)
    active_key_tier: Mapped[str] = mapped_column(String(20), default="free", server_default="free")

    # Relationships
    characters: Mapped[list["Character"]] = relationship(back_populates="owner")

    __table_args__ = (
        Index("idx_users_email", "email"),
        Index("idx_users_role", "role"),
        Index("idx_users_is_active", "is_active"),
    )
```

**Note on email index:** The `unique=True` on the column already creates a unique index. The explicit `idx_users_email` in `__table_args__` would be redundant. Use ONLY `unique=True` on the column definition (which creates the unique constraint/index), and add separate indexes only for `role` and `is_active` if query patterns warrant it.
