# Phase 999.1: Video Generation — Kie.ai Sora 2 - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Convert generated meme background images into 10-15 second portrait videos using Kie.ai's Sora 2 image-to-video API. Videos are generated on-demand (opt-in per content package), run asynchronously in the background, and use LLM-generated motion prompts that improve via manual feedback. Includes GCS upload for public image URLs, cost tracking with hard daily cap, and Sora 2 character registration for visual consistency.

</domain>

<decisions>
## Implementation Decisions

### Video Source Image
- **D-01:** Send the **raw background image** (Gemini-generated, no text, no watermark) to Sora 2. Video stays clean — text/legends are added later by FFmpeg in Phase 999.2. This keeps video quality high and avoids text distortion during animation.

### Motion Prompts
- **D-02:** Use **LLM-generated prompts** (Gemini) to create a unique motion prompt per video. The LLM receives image metadata (theme, pose, scene, phrase context) and generates a tailored motion description. Cost is negligible (~$0.001/prompt).
- **D-03:** **Manual feedback loop** for prompt improvement. User can rate videos (good/bad) and add notes. A `video_prompt_notes` field per theme accumulates learnings (e.g., "avoid fast camera movements", "more particle effects work well"). The LLM reads these notes when generating the next prompt for that theme.

### Public URL Strategy
- **D-04:** Use **Google Cloud Storage (GCS)** for image uploads. Leverage existing Google API credentials. Upload background image to GCS bucket, get a signed URL (1-hour expiry), pass to Kie.ai. Independent of Phase 14 (Cloudflare R2 is for Instagram CDN).

### Pipeline Integration
- **D-05:** **Opt-in per content package.** User selects which approved content packages get video generation from the gallery UI or API. No auto-generation — user controls cost. Videos are never generated automatically during pipeline runs.
- **D-06:** **Background async processing.** User triggers video gen, content package gets `video_status='generating'`, user continues working. Video appears when done (30-120s). Poll via existing job system or status endpoint.

### Sora 2 Model Tier
- **D-07:** Default to **Standard tier** (`sora-2-image-to-video` at $0.15/10s). 480-720p output is sufficient for Instagram Reels (compressed to 720p anyway). User can upgrade per-video if needed in the future.

### Video Duration
- **D-08:** **User selectable** — 10 or 15 seconds per video. Default to 10s. Simple toggle in the UI/API. 10s costs $0.15, 15s costs $0.225.

### Cost Budget Guardrails
- **D-09:** **Hard daily cap** via `VIDEO_DAILY_BUDGET_USD` env var (default $3.00 = ~20 standard videos/day). Once reached, video gen is blocked until next day. Show remaining budget in the UI before generating.

### Sora 2 Character Consistency
- **D-10:** **Register characters** with Kie.ai's `character_id_list` feature. Upload reference images for each character (Mago Mestre, future chars) to Kie.ai. Pass character IDs in every video generation request for visual consistency across videos.

### Claude's Discretion
- Error handling and retry strategy for failed Kie.ai tasks
- GCS bucket naming, lifecycle policies, and cleanup of uploaded images
- Exact Gemini prompt template for motion prompt generation
- Database schema details (column types, indexes)
- Async job polling interval and timeout values

</decisions>

## Specific Ideas

- Each character should be registered with Kie.ai for visual consistency (character_id_list)
- Motion prompts should reference the specific phrase context so the wizard's action relates to the meme content
- `video_prompt_notes` per theme allows iterative quality improvement without code changes
- Budget display should show "X of $3.00 remaining today" before user confirms video generation

</specifics>
