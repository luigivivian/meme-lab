# Phase 999.5: Interactive Reels Pipeline - Research

**Researched:** 2026-03-27
**Domain:** React stepper UI, step-based API design, pipeline state machine, video segmentation
**Confidence:** HIGH

## Summary

This phase transforms the existing fire-and-forget Reels pipeline (Phase 999.4) into an interactive step-by-step flow. The current pipeline runs all 5 steps sequentially in a background task with polling. The new design breaks this into 6 discrete steps, each requiring user approval before proceeding, with regenerate/edit options per step.

The frontend stack is already in place: Next.js 15, React 19, framer-motion 12.35.2, shadcn/ui components, SWR for data fetching. The backend uses FastAPI with async SQLAlchemy and background tasks. All 5 pipeline modules (image_gen, script_gen, tts, transcriber, video_builder) are already standalone and callable individually -- the orchestrator (`main.py`) just chains them. The refactoring is primarily about exposing each step as its own API endpoint and building a stepper UI that drives progression.

**Primary recommendation:** Build a stepper component using framer-motion AnimatePresence for step transitions, driven by a `step_state` JSON column on ReelsJob. Each step gets approve/regenerate/edit endpoints. The pipeline orchestrator is replaced by per-step API calls initiated by the frontend.

### Pattern 1: Stepper with framer-motion AnimatePresence
**What:** Each step renders inside `AnimatePresence` with slide animation based on direction.
**When to use:** Multi-step wizard with forward/back navigation.
**Example:**
```typescript
import { AnimatePresence, motion } from "framer-motion";

const variants = {
  enter: (direction: number) => ({ x: direction > 0 ? 300 : -300, opacity: 0 }),
  center: { x: 0, opacity: 1 },
  exit: (direction: number) => ({ x: direction > 0 ? -300 : 300, opacity: 0 }),
};

function StepContent({ step, direction }: { step: number; direction: number }) {
  return (
    <AnimatePresence mode="wait" custom={direction}>
      <motion.div
        key={step}
        custom={direction}
        variants={variants}
        initial="enter"
        animate="center"
        exit="exit"
        transition={{ duration: 0.3 }}
      >
        {/* Step content based on step number */}
      </motion.div>
    </AnimatePresence>
  );
}
```

### Pattern 2: Step-Based API Design
**What:** Each pipeline step gets its own endpoint trio: execute, approve, artifacts.
**When to use:** Interactive pipeline where each step requires user intervention.
**Example API surface:**
```
POST   /reels/{job_id}/step/prompt       # Save/update prompt text
POST   /reels/{job_id}/step/images       # Generate images (background task)
POST   /reels/{job_id}/step/script       # Generate script (background task)
POST   /reels/{job_id}/step/tts          # Generate TTS (background task)
POST   /reels/{job_id}/step/srt          # Generate SRT (background task)
POST   /reels/{job_id}/step/video        # Assemble final video (background task)

POST   /reels/{job_id}/approve/{step}    # Mark step as approved, advance current_step
POST   /reels/{job_id}/regenerate/{step} # Re-run step (keeps prior steps)
PUT    /reels/{job_id}/edit/{step}       # Update step artifacts (prompt text, script JSON, SRT)

GET    /reels/{job_id}/artifacts/{step}  # Get step artifacts (images, audio, srt, etc.)
GET    /reels/{job_id}/step-state        # Get full step_state JSON for resume
```

### Pattern 3: step_state JSON Column
**What:** Single JSON column persists the full state machine for each job.
**When to use:** When step progression needs persistence across sessions.

**SQLAlchemy change detection:** Use `dict.copy()` or `flag_modified()` when updating JSON columns (per Phase 19 pattern: "dict copy for SQLAlchemy JSON change detection").

### Pattern 4: Video Segmentation for Long Content
**What:** Videos >30s split into segments, each with own image set.
**When to use:** Per D-04, when target_duration > 30s.
**Strategy:**
1. Generate roteiro first (covers full video duration)
2. Split cenas into segments of ~30s each
3. Generate images per segment (same character refs/DNA for consistency)
4. Generate TTS for full narracao_completa (one audio file)
5. Generate SRT from full audio
6. Build segment videos (images + audio slice + SRT slice per segment)
7. Concatenate segments with xfade crossfade

**FFmpeg segment concat pattern:**
```python
# Build each segment as separate MP4
for i, segment in enumerate(segments):
    build_reel_video(
        image_paths=segment["images"],
        audio_path=segment["audio_slice"],
        srt_path=segment["srt_slice"],
        output_path=f"segment_{i}.mp4",
    )

# Concatenate with xfade between segments
# Chain xfade: offset = sum of prior segment durations - cumulative transition durations
cmd = ["ffmpeg", "-y"]
for seg_path in segment_paths:
    cmd += ["-i", seg_path]
# Build xfade filter chain...
```

### Anti-Patterns to Avoid
- **Monolithic step handler:** Do NOT create one giant endpoint that handles all steps. Each step should be its own endpoint for clarity.
- **Polling during step execution:** Use SWR refreshInterval (already established pattern) rather than WebSocket. The project has zero WebSocket infrastructure and adding it is overkill.
- **Re-running all steps on regenerate:** Per D-02, regenerating step N only re-runs step N. Prior steps remain intact. Subsequent steps may need to be re-run if their input changed (e.g., regenerating images invalidates script).
- **Complex state machine library:** Do NOT add XState or similar. The step_state JSON + current_step integer is sufficient for 6 linear steps.

## Common Pitfalls

### Pitfall 1: SQLAlchemy JSON Column Mutation Detection
**What goes wrong:** Updating a nested key in a JSON column does not trigger SQLAlchemy to detect the change.
**Why it happens:** SQLAlchemy tracks identity, not deep equality on JSON columns.
**How to avoid:** Use `from sqlalchemy.orm.attributes import flag_modified; flag_modified(job, "step_state")` after mutation, OR copy dict before modifying: `job.step_state = {**job.step_state, "key": new_value}`.
**Warning signs:** Step state appears to save but reverts on reload.

### Pitfall 2: Background Task DB Session Scoping
**What goes wrong:** Using the request's DB session in a background task fails because the session closes with the request.
**Why it happens:** FastAPI dependency injection scopes sessions to request lifecycle.
**How to avoid:** Use `get_session_factory()` pattern (already established in reels.py Phase 999.4).
**Warning signs:** "Session is closed" errors in background task logs.

### Pitfall 3: Audio File Serving for Frontend Preview
**What goes wrong:** Frontend cannot play audio preview because the file is on the server filesystem, not served via HTTP.
**Why it happens:** Generated artifacts are saved to `output/reels/{job_dir}/` but not exposed via API.
**How to avoid:** Add a `GET /reels/{job_id}/artifacts/{filename}` endpoint that serves files via `FileResponse`. Or use the existing proxy pattern (`/api/reels/...` -> FastAPI).
**Warning signs:** Audio/video preview returns 404 or CORS error.

### Pitfall 4: Step Invalidation on Regenerate
**What goes wrong:** User regenerates images but script still references old image indices.
**Why it happens:** Downstream steps depend on upstream artifacts.
**How to avoid:** When regenerating step N, mark all steps N+1 through 6 as `approved: false` and clear their artifacts. User must re-approve (or re-generate) downstream steps.
**Warning signs:** Script references non-existent image indices, SRT timings don't match audio.

### Pitfall 5: FFmpeg xfade Offset Calculation for Segments
**What goes wrong:** Concatenated segments have wrong timing, audio/video desync.
**Why it happens:** xfade offset must account for cumulative transition durations: `offset_i = sum(durations[0:i]) - i * transition_duration`.
**How to avoid:** Calculate offsets carefully using the formula. Test with 2 segments first before N segments.
**Warning signs:** Black frames between segments, audio drift.

### Pitfall 6: framer-motion AnimatePresence key Prop
**What goes wrong:** Step transitions don't animate, or all steps render simultaneously.
**Why it happens:** AnimatePresence requires a unique `key` prop on its direct child to trigger exit/enter animations.
**How to avoid:** Always use `<motion.div key={currentStep}>` as direct child of `<AnimatePresence>`.
**Warning signs:** Steps stack on top of each other, no exit animation.

const STEPS = [
  { id: "prompt", label: "Prompt", icon: MessageSquare },
  { id: "images", label: "Imagens", icon: Image },

  return (
    <div className="flex items-center gap-2">

        const isComplete = stepState[step.id]?.approved;
        return (
          <div key={step.id} className="flex items-center gap-2">
            <div className={cn(
              "flex items-center justify-center w-10 h-10 rounded-full border-2",
              isComplete && "bg-emerald-500 border-emerald-500",

### Audio Preview Component (Simple Play Button)
```typescript
function AudioPreview({ audioUrl }: { audioUrl: string }) {
  const [playing, setPlaying] = useState(false);
  const audioRef = useRef<HTMLAudioElement | null>(null);

  function toggle() {
    if (!audioRef.current) {
      audioRef.current = new Audio(audioUrl);
      audioRef.current.onended = () => setPlaying(false);
    }
    if (playing) {
      audioRef.current.pause();
    } else {
      audioRef.current.play();
    }
    setPlaying(!playing);
  }

  return (
    <Button variant="outline" size="sm" onClick={toggle}>
      {playing ? <Pause className="h-4 w-4 mr-2" /> : <Play className="h-4 w-4 mr-2" />}
      {playing ? "Pausar" : "Ouvir Preview"}
    </Button>
  );
}
```

### Step Approve/Regenerate API Pattern
```python
@router.post("/{job_id}/approve/{step}")
async def approve_step(
    job_id: str,
    step: str,
    current_user=Depends(get_current_user),
    db: AsyncSession = Depends(db_session),
):
    job = await _get_user_job(job_id, current_user.id, db)
    state = dict(job.step_state or {})
    if step not in state:
        raise HTTPException(400, f"Step '{step}' has no artifacts yet")
    state[step]["approved"] = True
    # Advance current_step
    step_order = ["prompt", "images", "script", "tts", "srt", "video"]
    idx = step_order.index(step)
    if idx + 1 < len(step_order):
        state["current_step"] = idx + 1
    job.step_state = state
    flag_modified(job, "step_state")
    await db.commit()
    return {"step": step, "approved": True, "current_step": state["current_step"]}
```

### Video Segmentation Logic
```python
def segment_roteiro(script: dict, max_segment_duration: float = 30.0) -> list[dict]:
    """Split roteiro cenas into segments of ~max_segment_duration seconds."""
    segments = []
    current_segment = {"cenas": [], "duration": 0.0}

    for cena in script["cenas"]:
        if current_segment["duration"] + cena["duracao_segundos"] > max_segment_duration and current_segment["cenas"]:
            segments.append(current_segment)
            current_segment = {"cenas": [], "duration": 0.0}
        current_segment["cenas"].append(cena)
        current_segment["duration"] += cena["duracao_segundos"]

    if current_segment["cenas"]:
        segments.append(current_segment)

    return segments
```

## State of the Art

| Old Approach (Phase 999.4) | New Approach (Phase 999.5) | Impact |
|---|---|---|
| Fire-and-forget: POST /generate runs all 5 steps | Step-by-step: user approves each step | Full control, can edit/regenerate per step |
| Single background task runs entire pipeline | Per-step background tasks (only generating step runs in background) | Faster feedback, resumable |

## Open Questions

1. **File serving for previews**
   - What we know: Generated artifacts live in `output/reels/{job_dir}/`. Frontend needs to preview images, audio, and video.
   - What's unclear: Whether to serve files via FastAPI FileResponse or copy to a static-served directory.
   - Recommendation: Add a `GET /reels/{job_id}/file/{filename}` endpoint returning FileResponse. Simplest approach, consistent with existing patterns.

2. **Individual image regeneration UX**
   - What we know: D-02 says "pode refazer 1 das N imagens sem perder as outras".
   - What's unclear: Whether regenerating one image should automatically invalidate the script step.
   - Recommendation: Regenerating any image marks the images step as "partially regenerated" and marks script/tts/srt/video as needing re-approval. User decides whether to re-generate downstream.

3. **Segment audio slicing**
   - What we know: TTS generates full narration audio. Segments need audio slices.
   - What's unclear: How to accurately split audio at segment boundaries.
   - Recommendation: Use SRT timestamps to determine segment audio boundaries, then `ffmpeg -ss start -to end -i audio.wav segment_audio.wav` for precise slicing.
